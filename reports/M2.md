## 1. Data Model

**Relational.** All data is stored in SQLite as two fact tables:

```sql
ratings(user_id INTEGER, movie_id INTEGER, rating REAL, timestamp INTEGER,
        PRIMARY KEY (user_id, movie_id))

movies(movie_id INTEGER PRIMARY KEY, title TEXT)
```

Rationale: provenance queries are naturally expressed as SQL set operations (SELECT, DELETE). A relational model lets us define provenance as a subset of tuples and measure it precisely, without needing a graph or document store.

## 3. Parser (`src/load.py`)

`load.py` performs three steps:

1. **Open** `data/raw/ml-100k/u.data` (tab-separated: `user_id movie_id rating timestamp`)
2. **Batch insert** all 100,000 rows into `ratings` via `INSERT OR REPLACE`
3. **Parse** `u.item` (pipe-separated, latin-1 encoding) and insert movie titles into `movies`

No external libraries beyond Python's built-in `csv` and `sqlite3`. Running `python src/load.py` from the project root creates `data/ratings.db` in < 0.5 s.

## 4. Auxiliary Structure: Inverted Index

**Motivation:** A naive why-provenance algorithm must test all raters of a movie. For a movie with 300+ raters, this means hundreds of CF re-evaluations. The inverted index pre-ranks raters by contribution so we only examine the top-C.

**Structure:** Table `movie_contributors(movie_id, user_id, weight)` where `weight = rating` (the rating value is a proxy for how strongly a user endorsed a movie and thus contributed to its CF score). Indexed on `movie_id` for O(1) lookup.

**Pseudo-code:**

```
ALGORITHM BuildInvertedIndex(ratings R, top_C = 50)
1.  FOR each movie m in R:
2.      contributors ← {(u, rating) | (u, m, rating) ∈ R}
3.      top_contributors ← top-C entries in contributors sorted by rating DESC
4.      INSERT (m, u, rating) INTO movie_contributors FOR each (u, rating) IN top_contributors
5.  CREATE INDEX ON movie_contributors(movie_id)
```

**Statistics:**

| Metric | Value |
|---|---|
| # movies indexed | 1,682 |
| # index entries | 46,435 |
| Build time | 0.16 s |
| Storage overhead | < 2× raw ratings table |

## 5. Plan for M3

**WhyProv:** Given `(user u, movie m, k)`, use the inverted index to retrieve the top-C raters of `m`. Greedily remove raters one by one (highest-weight first) and re-estimate CF score until `m` drops out of `u`'s top-k. The returned set is the witness (why-provenance).

**QueryRewrite:** Given `(user u, target movie t)` where `t ∉ TopK(u, k)`, identify users whose high ratings of similar movies would most increase `t`'s predicted score for `u`. Return the minimal single-rating addition that surfaces `t` into top-k.

Both algorithms share the same index; the key difference from the baseline is that only the top-C contributors are ever examined, reducing search space from O(|R|) to O(C) per query.

## Part 1 · Project Description

### Problem Statement

This project addresses **query provenance for explainable recommendations**.

**Setting:** A collaborative-filtering (CF) model trained on MovieLens 100K produces a top-k recommendation list for each user. The data is stored in a relational table `ratings(user_id, movie_id, rating, timestamp)`.

**Two queries we answer:**

1. **Why-provenance query:** Given user `u` and movie `m` that appears in `u`'s top-k, return the *minimal* subset of ratings in the database that, if removed, would cause `m` to drop out of the top-k. This subset is called the *why-provenance* of `m` for `u`.

2. **Query-rewrite query:** Given user `u` and a target movie `t` that is *not* in `u`'s top-k, find the *smallest* edit to the database (adding or removing ratings) that makes `t` appear in the top-k.

**Formal query class:**
- Input: `(u: user_id, m: movie_id, k: int)` where `u ∈ [1,943]`, `k ∈ [1,20]`, `m ∈ TopK(u,k)`
- Output: a minimal witness set `W ⊆ ratings` s.t. `m ∉ TopK_without_W(u,k)`

### Why It Matters

CF models are black boxes: users and product managers cannot understand *why* a movie was recommended. Provenance provides a principled, data-level explanation without modifying the CF model. This is directly relevant to **explainable AI** and **responsible product design** — both growing regulatory and user-trust concerns in recommendation systems.

### Why It's Hard

- **Naive why-provenance** requires removing each rating subset and re-running CF — combinatorial in the number of ratings.
- **Naive query rewrite** is combinatorial in possible edits.
- Re-running CF is expensive: even on 100K ratings, SVD decomposition takes seconds.

The core challenge is **pruning the search space** so that provenance can be computed in milliseconds.

### Goal

Design and implement:
1. An **inverted index** from `movie_id → contributing users`, built once at index time.
2. A **WhyProv algorithm** that uses the index to prune: instead of scanning all 100K ratings, only examine the top-C contributing users for movie `m`.
3. A **QueryRewrite algorithm** that identifies the minimal rating edit via the same index.
4. **Experimental validation**: show ≥5× speedup over naive full-scan baseline, across ≥50 random (user, movie) pairs.

## Part 2 · Datasets & Tools

### Dataset

- **Name:** MovieLens 100K
- **Link:** https://files.grouplens.org/datasets/movielens/ml-100k.zip
- **Size:** ~5 MB, 100,000 ratings, 943 users, 1,682 movies, ratings 1–5
- **Schema:** `u.data` → tab-separated `(user_id, movie_id, rating, timestamp)`
- **Split used:** full `u.data` (no train/test split needed — provenance operates on the full database)

### Tools

| Tool | Role |
|---|---|
| Python 3.11 | implementation language |
| SQLite | relational storage, index construction |
| `scikit-surprise` | CF model (SVD) — used as a black box, not reimplemented |
| `numpy` | similarity computations |
| `matplotlib` | runtime/size/quality figures |

### Data Model

Relational. Single fact table:
```sql
ratings(user_id INTEGER, movie_id INTEGER, rating REAL, timestamp INTEGER,
        PRIMARY KEY (user_id, movie_id))
```

Inverted index table (to be built in M2):
```sql
movie_contributors(movie_id INTEGER, user_id INTEGER, weight REAL)
-- weight = contribution of user_id's ratings to movie_id's CF score for any query user
```
